
# AUTOMATION DESIGN

## 1. Approach

To automate recurring business reports, I would build a scheduled data automation pipeline that runs SQL queries and generates reports automatically.

Each report will be executed by a scheduled job that triggers the Go automation tool developed in Part 2. The tool connects to PostgreSQL, executes predefined analytical queries, and exports the results as JSON files or sends them to downstream systems.

The automation process will include:

1. Scheduled execution using a job scheduler.
2. Running SQL queries for each business report.
3. Exporting the results to structured output (JSON/CSV).
4. Sending notifications if thresholds or alerts are detected.

Example automation tasks:

| Report | Frequency | Output |
|------|------|------|
| Weekly Sales Report | Every Monday | JSON report |
| Low Stock Alerts | Daily | Alert notification |
| Customer Churn Risk | Daily | Customer list |
| Monthly Revenue by Category | Monthly | Aggregated report |

This approach removes manual work and ensures reports are generated consistently and on time.

## 2. Architecture

A simple architecture for this automation system is shown below.

Scheduler (Cron / Airflow)
        |
        v
Go Automation Tool
        |
        v
PostgreSQL Database
        |
        v
Report Output / Notification System
(JSON files / Email / API)

### Workflow

1. The scheduler triggers the Go automation tool.
2. The Go application executes the required SQL query.
3. The results are exported to JSON files or sent to external services.
4. Alerts are generated if specific conditions are met (e.g., low stock).

Example pseudocode:

scheduler triggers job
    |
    run Go CLI
        |
        connect to PostgreSQL
        execute query
        export results
        send notification

This architecture is simple, reliable, and easy to scale.

## 3. Technology Choices

The following technologies are selected for automation.

### Scheduler

Possible options:

- Cron (simple environments)
- Apache Airflow (data pipeline orchestration)
- Kubernetes CronJob (containerized environments)

For this project, **Cron** is sufficient because the automation tasks are straightforward and time-based.

Example:

0 8 * * MON run weekly_sales_report  
0 7 * * * run low_stock_alert

### Data Processing

- **Go** for the automation tool
- PostgreSQL for data storage
- SQL queries for analytics

Go is chosen because it is efficient, lightweight, and well-suited for CLI automation tools.

### Notification / Output

Possible integrations:

- Email alerts
- Slack notifications
- REST API push to BI platform
- JSON/CSV reports

For example:

- Low stock alerts → send notification
- Sales report → export JSON file

Reasons:

3. It is relatively simple to implement.

## 5. Scalability

If the system needs to handle **10 concurrent automated reports**, the architecture can scale in several ways.

### Parallel Execution

The Go automation tool can run multiple report jobs concurrently using goroutines.

Example concept:

go run report_1  
go run report_2  
go run report_3

This allows multiple reports to be generated at the same time.

### Job Queue

For larger workloads, a job queue system can be introduced.

Possible tools:

- Redis Queue
- RabbitMQ
- Kafka

The workflow would be:

Scheduler
   |
   v
Queue
   |
   v
Worker Services (Go)

Workers consume jobs from the queue and execute queries in parallel.

### Database Optimization

To support concurrent reporting:

- Use proper indexing
- Use read replicas for analytical queries
- Optimize SQL queries with window functions and CTEs

This ensures the system can handle multiple report requests without affecting operational workloads.

## Conclusion
