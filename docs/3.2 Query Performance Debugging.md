
# 3.2 Query Performance Debugging

A business user reports that the following query takes **45 seconds** to execute:

```sql
SELECT
    c.name,
    c.email,
    COUNT(o.id) as order_count,
    SUM(o.total_amount) as total_revenue
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE o.order_date >= '2024-01-01'
GROUP BY c.id, c.name, c.email
HAVING COUNT(o.id) > 5
ORDER BY total_revenue DESC;
```

Database size:

- 2 million customers  
- 15 million orders  
- Only primary key indexes exist  

---

## 1. Root Cause

The query is slow mainly due to **large table scans and missing indexes**.

### Full Table Scan

The filter condition:

```sql
o.order_date >= '2024-01-01'
```

has no supporting index. PostgreSQL must scan a large portion of the **15 million rows in the orders table** to evaluate this condition.

### Expensive Join

The query performs a join between:

- `customers` (2M rows)
- `orders` (15M rows)

Without an index on `orders.customer_id`, PostgreSQL cannot efficiently locate orders belonging to each customer.

### Inefficient LEFT JOIN

Although a LEFT JOIN is used, the condition:

```sql
WHERE o.order_date >= '2024-01-01'
```

forces PostgreSQL to behave like an **INNER JOIN**, but the optimizer may still process it less efficiently.

### Heavy Aggregation

The query performs aggregation operations:

- `COUNT(o.id)`
- `SUM(o.total_amount)`

over millions of rows before applying the `HAVING` filter.

---

## 2. Quick Fixes

Several immediate improvements can significantly reduce query execution time.

### Add Index on orders.customer_id

```sql
CREATE INDEX idx_orders_customer_id
ON orders(customer_id);
```

This speeds up the join between customers and orders.

### Add Index on orders.order_date

```sql
CREATE INDEX idx_orders_order_date
ON orders(order_date);
```

This allows PostgreSQL to quickly filter orders from 2024 onward.

### Composite Index (Best Immediate Improvement)

```sql
CREATE INDEX idx_orders_customer_date
ON orders(customer_id, order_date);
```

This index helps both:

- joining orders to customers
- filtering by date

---

## 3. Long-term Solutions

For sustained performance at scale, structural improvements should be implemented.

### Table Partitioning

Partition the `orders` table by `order_date` (monthly or yearly).

Benefits:

- Queries scanning recent data only read relevant partitions
- Smaller indexes per partition
- Faster maintenance operations

Example:

```sql
PARTITION BY RANGE (order_date)
```

### Pre-Aggregated Reporting Tables

For frequently requested reports, create summary tables such as:

```
customer_order_summary
```

Example fields:

- customer_id
- order_count
- total_revenue
- last_order_date

This reduces the need to scan millions of rows repeatedly.

### Read Replica for Analytics

Analytical queries should run on a **read replica database** to avoid impacting transactional workloads.

---

## 4. Optimized Query Rewrite

A more efficient approach is to filter and aggregate orders **before joining to customers**.

```sql
WITH customer_orders AS (
    SELECT
        customer_id,
        COUNT(id) AS order_count,
        SUM(total_amount) AS total_revenue
    FROM orders
    WHERE order_date >= '2024-01-01'
    GROUP BY customer_id
    HAVING COUNT(id) > 5
)

SELECT
    c.name,
    c.email,
    co.order_count,
    co.total_revenue
FROM customer_orders co
JOIN customers c
    ON c.id = co.customer_id
ORDER BY co.total_revenue DESC;
```

---

## Why This Version Is Faster

This optimized version improves performance by:

1. **Filtering early** – Orders are filtered before the join occurs.  
2. **Reducing join size** – Instead of joining 15 million orders, the query joins only aggregated customer results.  
3. **Smaller aggregation scope** – Aggregation happens on a filtered dataset.  
4. **Better use of indexes** – With indexes on `customer_id` and `order_date`, PostgreSQL can efficiently scan relevant rows.

---

## Expected Impact

With proper indexing and query restructuring, execution time could drop from **~45 seconds to around 1–2 seconds**, depending on hardware and data distribution.
